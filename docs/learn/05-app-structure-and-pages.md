# 05 项目目录与页面骨架：为什么先搭空页面也有价值

## 本次目标

这一步做的不是完整业务，而是先把项目的页面结构和目录结构站稳。

很多人会觉得“空页面没意义”。其实不是。  
对于中后台项目，先搭页面骨架有两个很重要的价值：

1. 你能更早验证信息架构是否合理
2. 你能给后续业务开发留出稳定位置

## 为什么目录这么拆

## 1. `src/app`

这是 Next.js App Router 的页面入口目录。

它负责：

- 路由
- 页面布局
- 页面级组件组织

你现在看到的 `(dashboard)` 和 `(auth)`，是路由分组。

### 为什么要用路由分组

因为登录页和后台页通常长得不一样。

- 登录页一般是独立布局
- 后台页一般有侧边栏、顶部区域、内容区

所以把它们分开，是为了让布局职责清楚。

## 2. `src/components`

这里放可复用组件，而不是页面本身。

继续细分的意义是：

- `ui` 放通用 UI 组件
- `common` 放业务无关但项目常用的组件
- `layout` 放整体页面壳子

这样你后面组件一多，不会全堆在一个目录里。

## 3. `src/lib`

这里放工具层和基础能力，比如：

- `utils.ts`
- `prisma.ts`
- `validations`

简单理解就是：这里放“不是页面，但很多地方都会依赖”的代码。

## 4. `src/stores`

这里是 Zustand 的位置。

虽然现在只放了一个简单的预约筛选 store，但这是在提前告诉项目结构：

- 前端状态不要和页面 JSX 混在一起

## 5. `src/types`

这里放领域类型和常量定义。

这样做的好处是：

- 页面不用自己硬编码业务枚举
- 类型和页面逻辑解耦

## 当前页面骨架各自的作用

## 1. `/`

根路由现在的作用非常简单：重定向到 `/dashboard`。

### 为什么这样做

因为这个项目的核心入口是后台，而不是官网首页。  
这符合你项目的定位：**中后台系统，不是展示站。**

## 2. `/login`

当前还是一个演示登录页，没有正式接 Auth.js 登录流程。

### 为什么先保留它

因为它能先把“认证入口”这个概念占住，后面接真实登录时就不用再改整体路由结构。

## 3. `/dashboard`

工作台页面目前展示的是：

- 今日预约
- 待确认
- 今日完成
- 待回访

以及当前开发顺序和演示账号。

### 为什么工作台先放这些信息

因为这些数据最能体现项目主线：

- 预约
- 履约
- 回访

## 4. `/services`

现在是服务项目列表的静态骨架页。

它的目标不是完成 CRUD，而是提前固定：

- 页头结构
- 表格布局
- 新增按钮位置

后续接真实数据时，只需要替换数据来源，不用重想页面结构。

## 5. `/customers`

当前是客户和宠物概览骨架页。

它提前验证了这件事：

- 客户信息和宠物信息应当一起展示，但不能混成一团

## 6. `/appointments`

它是当前最值得注意的页面骨架，因为这里已经提前放了：

- 预约筛选
- 预约状态选项
- 列表结构

这相当于在为后面的预约主链路提前铺路。

## 为什么我提前补了这些基础代码

## 1. `app-shell.tsx`

它是后台布局壳子，负责：

- 侧边栏
- 主内容区域
- 菜单高亮

如果没有这个文件，每个后台页面都要重复写一遍布局。

## 2. `page-header.tsx`

它统一了页面顶部结构：

- 标题
- 描述
- 右侧操作按钮

中后台项目里这类结构重复率很高，早点抽出来很划算。

## 3. `button.tsx`

这是你后面会频繁复用的基础按钮组件。

虽然现在只是基础版，但它已经具备了：

- 变体
- 尺寸
- `asChild`

这为后面更完整接入 shadcn/ui 打了基础。

## 4. `prisma.ts`

这是 Prisma Client 的统一出口。

### 为什么不能到处 `new PrismaClient()`

因为在开发环境下如果反复创建实例，很容易产生连接问题。  
所以通常会封装成一个全局复用版本。

## 5. `appointment-filter.store.ts`

这是一个很小的 Zustand store，但它很有教学价值。

它在告诉你：  
**筛选状态属于前端 UI 状态，不一定非要和页面代码绑死。**

## 你自己如何复现

### 第一步：先想页面分组，而不是先写某个页面

先判断：

- 哪些页面属于认证区
- 哪些页面属于后台区

### 第二步：先搭布局，再写页面内容

优先完成：

- 根布局
- 后台布局壳子
- 公共页头

### 第三步：先写骨架页，再接数据

尤其是中后台项目，这种顺序很适合初学者：

1. 先固定页面框架
2. 再放假数据
3. 再接真实数据库
4. 最后补表单和交互

### 第四步：把“公共能力”尽早抽出来

比如：

- 按钮
- 页面标题
- 布局壳子
- 工具函数

这些抽得越晚，后面改动范围越大。

---

## 容易踩坑的点

## 1. 只按页面数量搭结构，不按职责拆目录

一开始看不出问题，后面文件一多就会乱。

## 2. 过早把业务逻辑塞进 page 文件

`page.tsx` 应该承担页面组织职责，而不是把所有逻辑都堆进去。

## 3. 还没想清楚布局差异，就急着写所有页面

这会导致登录页和后台页的结构互相污染。

## 4. 认为假数据页面没价值

事实上，假数据页面可以提前暴露很多信息架构问题。

看完后你要能自己回答：

1. 为什么 `src/app`、`src/components`、`src/lib` 要分开？
2. 为什么登录区和后台区值得用路由分组拆开？
3. 为什么先做页面骨架不是浪费时间？
4. 为什么 `prisma.ts` 和 `page-header.tsx` 这类基础文件值得尽早抽出来？

如果这些你能讲清楚，说明你已经开始用“工程视角”看项目，而不是只盯着单个页面。
