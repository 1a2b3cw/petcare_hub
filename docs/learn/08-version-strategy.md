# 08 版本策略：为什么不要把项目长期写成 `latest`

## 本次目标

这一篇解决的是一个很常见、但很容易被忽略的问题：

**项目现在能跑，不代表下周、下个月、换台电脑以后还一定能跑。**

如果版本策略没定好，最典型的后果就是：

- 你今天 `npm install` 没问题
- 几周后重新安装依赖，突然报错
- 教学文档和代码对不上
- 你自己都复现不出当时的环境

所以这一步做的事情不是“改几个版本号”，而是把项目从“能跑”推进到“可复现”。

---

## 这次我改了什么

我主要做了三件事：

### 1. 把 `package.json` 里大量 `latest` 改成了精确版本

比如从：

```json
"next": "latest"
```

改成：

```json
"next": "16.2.4"
```

这意味着你下次安装时，拿到的是明确版本，而不是“安装当时 npm 上最新的版本”。

### 2. 在 `package.json` 里加了 `engines`

这样项目会明确告诉你：

- 建议用哪个 Node 大版本
- 建议用哪个 npm 大版本

### 3. 添加了 `.nvmrc`

这个文件进一步把 Node 版本固定成了：

```txt
24.14.0
```

这样你以后切环境时，更容易知道该切到哪个 Node 版本。

---

## 为什么 `latest` 不适合长期保留

## 1. `latest` 会随着时间漂移

你今天执行安装命令时，`latest` 指向的是今天最新版本。  
过一段时间，它就会变成另一个版本。

这会导致一个问题：

**同一份代码，不同时间安装，可能拿到不同依赖环境。**

对学习项目来说，这非常不利，因为你会分不清：

- 是你代码写错了
- 还是依赖版本变了

## 2. 你这个项目还要配合教学文档一起使用

既然你希望后面能靠文档复现项目，那版本就不能漂。

否则文档里写的是一套环境，未来安装下来是另一套环境，就会出现：

- 代码对不上
- 行为不一致
- 报错位置不同

## 3. 中后台项目更怕“边开发边漂版本”

你这个项目后面还会接：

- Prisma
- Auth
- React Hook Form
- Zod

这些东西一旦版本漂移，报错和行为差异都可能比较明显。

---

## 当前采用的版本策略是什么

当前我给你定的是这套策略：

## 1. 直接依赖使用精确版本

也就是：

- `next`
- `react`
- `prisma`
- `zod`

这些你在 `package.json` 里直接声明的包，全部写成明确版本号。

### 为什么这样做

因为这些包最直接影响项目行为。

## 2. 间接依赖交给 `package-lock.json`

你不需要手动管理每个子依赖。  
这部分交给 npm 的锁文件处理。

### 为什么这样做

因为手动控制所有依赖树没有必要，成本太高。  
而 `package-lock.json` 已经能很好地解决“间接依赖锁定”问题。

## 3. Node/npm 用 `engines` + `.nvmrc` 约束

这能解决“同样项目，不同 Node 版本跑出来完全不一样”的问题。

---

## 为什么不是所有依赖都用 `^`

很多教程会写：

```json
"next": "^16.2.4"
```

这表示允许小版本自动升级。

这种策略在团队项目里不一定错，但你当前这个项目更适合：

**优先稳定、优先复现、优先教学一致性。**

所以你现在阶段更适合：

```json
"next": "16.2.4"
```

也就是精确锁定。

---

## `engines` 和 `.nvmrc` 分别有什么用

## `engines`

它写在 `package.json` 里，作用是告诉开发者和工具：

- 推荐的 Node 版本范围
- 推荐的 npm 版本范围

你可以把它理解为“项目级版本说明”。

## `.nvmrc`

它更偏工具使用场景，作用是告诉你：

- 这个项目建议直接用哪个 Node 版本

你可以把它理解为“快速切换环境的提示文件”。

---

## 为什么这一步对你特别重要

因为你不是只想把项目做出来，你还想：

1. 自己后面能复现
2. 通过教学文档重新搭建
3. 让项目长期保持可维护

那版本策略就不能靠运气。

如果你现在把版本定稳，后面你做：

- 服务项目 CRUD
- 客户/宠物档案
- 预约流转

就不会总被“依赖又变了”这种问题打断。

---

## 你以后什么时候可以升级版本

可以升级，但不要乱升。

建议你只在下面几种情况升级：

1. 你明确需要某个新能力
2. 当前版本有明显 bug 或安全问题
3. 你准备专门花一点时间处理升级兼容

### 正确姿势

不是顺手把所有包都升到最新，而是：

1. 明确要升级哪个包
2. 看 changelog
3. 升级后跑 `lint`、`build`
4. 记录这次升级影响

---

## 容易踩坑的点

## 1. 以为 `latest` 最省事

短期看省事，长期最容易让项目失控。

## 2. 只锁依赖版本，不管 Node 版本

这也不够。  
因为很多兼容性问题根本不是包版本问题，而是 Node 版本问题。

## 3. 升级依赖后不做验证

每次重要升级后，至少要跑：

```bash
npm run lint
npm run build
```

## 4. 把“稳定”理解成“永远不升级”

不是。  
正确理解是：**升级要有意识，不能漂。**

1. 为什么求职作品项目不适合长期保留 `latest`？
2. 为什么“直接依赖精确版本 + 锁文件锁间接依赖”是合理策略？
3. `engines` 和 `.nvmrc` 的作用分别是什么？
4. 以后什么时候应该升级依赖？

如果这几个问题你能讲清楚，说明你已经开始用“项目维护”的视角看版本，而不是只把版本号当成一串数字。

## 当前项目最终采用的版本基线

经过实际检查当前机器环境后，这个项目最终采用的是：

- Node：`24.14.0`
- npm：`11.9.0`

### 为什么不是一开始看到的 `22.22.0`

因为在 Cursor 终端里，直接输入 `node -v` 时，优先命中的是 Cursor 自带的辅助 `node.exe`。  
但 `npm.cmd` 实际走的是 NVM 管理的 `D:\nvm\nodejs\node.exe`。

项目真正要对齐的，是 **npm 和项目脚本实际执行时使用的 Node 版本**。  
所以最终我们把版本策略统一到了：

- `.nvmrc`：`24.14.0`
- `engines.node`：`>=24.14.0 <25`
- `engines.npm`：`>=11.9.0 <12`

这也是为什么版本策略不能只看一条 `node -v` 输出，而要看“项目脚本到底跑在谁身上”。
