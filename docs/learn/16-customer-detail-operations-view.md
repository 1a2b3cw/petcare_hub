# 第 16 步：客户详情页的运营视图

## 本次目标

把客户详情页从“只看档案和预约”升级成“客户运营视图”，让这个页面能直接回答：

- 这位客户最近有哪些预约和履约信息？
- 这位客户当前有哪些待处理回访？
- 这位客户手里有哪些优惠券？

做到这一步后，客户详情页就不再只是资料页，而是开始具备“客户 360 视图”的味道。

## 我做了什么

这一轮主要修改了：

```text
src/app/(dashboard)/customers/[id]/page.tsx
```

同时复用了前面已经写好的运营模块 action：

- `completeFollowUpTaskAction`
- `skipFollowUpTaskAction`
- `markCouponUsedAction`
- `markCouponExpiredAction`

## 这一步新增了哪些能力

### 1. 客户详情页增加“客户运营”概览

页面现在会直接展示：

- 待回访数量
- 未使用优惠券数量
- 最近一次建议回店日期

这让你打开客户详情页时，不用再切到运营页才能判断这个客户当前的复购状态。

### 2. 客户详情页展示回访任务

现在客户详情页里会列出这位客户最近的回访任务，并且如果任务状态还是 `PENDING`，可以直接：

- 标记完成
- 跳过

也就是说，客户详情页已经具备了最基本的运营处理能力。

### 3. 客户详情页展示优惠券

页面会展示这位客户最近的优惠券，包括：

- 券标题
- 优惠值
- 使用门槛
- 有效期
- 当前状态

如果优惠券还没用，还可以直接：

- 标记已用
- 标记过期

## 为什么这样做

## 1. 为什么要把运营信息放进客户详情页？

因为真实门店后台里，很多运营动作并不是在一个纯“运营页面”里完成的，而是穿插在查看客户详情时一起完成的。

比如店员点开一个客户时，最想立刻知道的是：

- 这个人最近有没有来过
- 该不该联系
- 手里有没有券

如果这些信息都藏在别的页面里，实际使用就会很割裂。

## 2. 为什么复用运营页的 Server Action，而不是再写一套？

因为“完成回访任务”“跳过回访任务”“标记优惠券状态”这些动作本质上和页面无关。

无论你是在：

- 运营页里点
- 客户详情页里点

它们做的都是同一件事。

所以最合理的做法就是复用已有 action，而不是复制一份逻辑。

这也是一个很典型的工程思路：**把业务动作抽成通用服务端动作，页面只负责入口和展示。**

## 关键文件说明

### `src/app/(dashboard)/customers/[id]/page.tsx`

这一轮这里新增了 3 类东西：

1. **查询层扩展**
   在原来的 `findUnique` 里继续 `include`：
   - `coupons`
   - `followUpTasks`
   - `appointments.visitRecord`

2. **展示层增强**
   在页面上新增：
   - 客户运营概览卡
   - 回访任务区块
   - 客户优惠券区块

3. **动作层接入**
   直接把运营模块的 action 绑定到按钮上，让客户详情页也能处理这些动作。

## 你自己如何复现

如果你想自己重做这一轮，推荐顺序是：

1. 先在客户详情页的 Prisma 查询里补 `coupons` 和 `followUpTasks`
2. 再补一个运营概览卡片
3. 再补回访任务列表
4. 最后补优惠券列表和状态按钮

## 验证方式

建议这样测：

1. 找一个已经有回访任务和优惠券的客户
2. 打开该客户详情页
3. 看是否能看到：
   - 待回访数量
   - 未使用优惠券数量
   - 最近建议回店时间
4. 在详情页直接点：
   - 标记回访完成
   - 标记优惠券已用 / 过期
5. 刷新页面，确认状态是否即时变化

## 容易踩坑的点

1. **只把列表搬过来，不补顶部概览**
   那客户详情页就还是“堆信息”，不是“帮助判断”的运营视图。

2. **重复写一套 action**
   这样后面很容易出现一个页面改了逻辑、另一个页面忘了同步的问题。

3. **忘了把 `appointments.visitRecord` 一起查出来**
   那“最近建议回店日期”就没法直接从客户详情页算出来。

## 做完以后项目状态

现在客户详情页已经从“档案页”升级成了“客户运营页的局部视图”。

这一步做完后，你的系统开始更像真实业务后台，而不是单纯的模块拼接：

- 运营页做集中处理
- 客户详情页做单客户聚合查看

这两个视角是互补的。
