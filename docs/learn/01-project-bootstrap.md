# 01 项目初始化：从空目录到可运行骨架

## 本次目标

这一步的目标很简单：先把项目从“只有需求文档的空目录”变成“可以继续开发的正式工程”。

如果没有这一步，后面所有业务开发都会没有落点。你就算想先写页面，也会立刻遇到这些问题：

- 没有 `package.json`，项目不能安装依赖
- 没有 Next.js 目录结构，页面没有地方写
- 没有基础脚本，项目不能启动、构建、校验
- 没有 Prisma 目录，数据库模型无处安放

所以这一步本质上是在做：**把需求变成工程。**

---

## 我做了什么

### 1. 确认当前目录状态

先确认工作区基本还是一个空目录，只有我们前面写好的文档，没有正式前端项目。

### 2. 尝试用 `create-next-app` 直接初始化

一开始尝试直接在当前目录执行 Next.js 官方初始化命令，但失败了。原因不是命令错，而是因为当前目录里已经存在这几个文件：

- `PROJECT_PRD.md`
- `DATABASE_DESIGN.md`
- `PRISMA_SCHEMA_DRAFT.prisma`

`create-next-app` 默认要求目录足够“干净”，否则它会担心覆盖已有文件。

### 3. 改为手工初始化

既然自动脚手架被现有文件挡住，那就直接手工搭骨架。这种做法完全没问题，而且对学习更有帮助，因为你能真正理解每个配置文件的作用。

我手工补了这些基础文件：

- `package.json`
- `tsconfig.json`
- `next-env.d.ts`
- `next.config.ts`
- `postcss.config.mjs`
- `eslint.config.mjs`
- `.gitignore`
- `.env.example`
- `.env`
- `components.json`

### 4. 安装依赖

然后执行 `npm install`，把 Next.js、React、TypeScript、Tailwind、Prisma、Zod 等依赖真正装进来。

中途第一次安装因为网络问题失败，报的是 `ECONNRESET`。这类问题很常见，不是代码错误，重新执行一次安装就成功了。

### 5. 建立 `src` 和 `prisma` 为核心的工程结构

完成安装后，再把项目目录真正拆出来，后续开发就有稳定落点。

---

## 为什么这样做

## 1. 为什么不重新开一个子目录

可以这么做，但我没有这么做，原因是你当前的需求文档和设计文档已经在这个根目录里了。如果再套一层子目录：

- 路径会变深
- 文档和代码会分散
- 后面你自己回看时更容易乱

对作品集项目来说，当前这种“根目录既有文档又有代码”的形式是合理的。

## 2. 为什么手工搭建反而更适合学习

自动脚手架适合快速开始，但它会帮你隐藏很多细节。你现在的目标不是只把项目跑起来，而是要通过这个项目提升工程理解。所以手工搭骨架有两个优点：

- 你知道项目为什么能运行
- 你知道每个文件缺失时会出现什么问题

## 3. 为什么先搭工程，不先写业务页面

很多初学者会急着先写页面，但真实开发里更合理的顺序是：

1. 先有工程底座
2. 再有数据模型
3. 再接业务模块

因为页面只是表现层，真正决定项目是否能持续推进的，是工程结构是否稳定。

## 关键文件说明

### `package.json`

这是项目的入口配置文件，负责两件事：

- 声明依赖
- 提供脚本命令

比如你现在能执行：

- `npm run dev`
- `npm run build`
- `npm run lint`
- `npm run prisma:generate`

都依赖这个文件。

### `tsconfig.json`

这是 TypeScript 配置文件，用来告诉编译器：

- 代码用什么模块系统
- 路径别名怎么解析
- 哪些文件需要被纳入类型检查

### `.env.example` 和 `.env`

这两个文件负责环境变量。

- `.env.example` 用来告诉未来的你或别人，这个项目需要哪些环境变量
- `.env` 是你本地真实运行时读取的值

### `.gitignore`

用来忽略不应该提交的内容，比如：

- `node_modules`
- `.next`
- `.env`

## 你自己如何复现

如果让你从零重新搭一次，推荐顺序如下：

### 第一步：准备一个空目录

目录里可以先放需求文档，但要注意如果你想用 `create-next-app` 直接初始化，目录最好尽量干净。

### 第二步：创建 `package.json`

至少先定义：

- `dev`
- `build`
- `start`
- `lint`

以及你项目需要的核心依赖。

### 第三步：安装依赖

先装这一批最核心的：

- `next`
- `react`
- `react-dom`
- `typescript`
- `eslint`
- `tailwindcss`
- `prisma`
- `@prisma/client`

### 第四步：补基础配置文件

把 Next.js、TypeScript、PostCSS、ESLint、环境变量这些配置补齐。

### 第五步：创建目录结构

至少创建：

- `src/app`
- `src/components`
- `src/lib`
- `src/types`
- `src/stores`
- `prisma`

### 第六步：做一次完整验证

至少确保下面三件事能成功：

- 依赖安装成功
- 项目可构建
- 类型检查和 lint 不报错

## 容易踩坑的点

## 1. 以为初始化失败就是项目做不下去

不是。像这次 `create-next-app` 初始化失败，并不是项目不能做，而是脚手架的目录限制。你要学会判断：

- 是工程逻辑错了
- 还是工具使用场景不匹配

## 2. 目录结构还没定，就开始狂写页面

这会导致后面重构成本非常高。尤其是中后台项目，业务模块多，结构如果一开始乱了，后面会很难收拾。

## 3. 忽视环境变量文件

很多人只写 `.env`，不写 `.env.example`。结果过几周自己都忘了项目需要哪些变量。

## 4. 把“能运行”和“可维护”混为一谈

项目能跑起来只是第一步。你做的是求职作品，不只是 demo，所以从一开始就要让项目结构清楚。

---

## 这一篇你应该学会什么

看完这篇，你至少要能回答：

1. 为什么当前项目没有用 `create-next-app` 直接初始化成功？
2. 为什么手工初始化并不等于“乱来”？
3. 一个 Next.js 项目最基础的入口文件有哪些？
4. 为什么正式开发前一定要先有工程骨架？

如果这 4 个问题你都能自己讲明白，这一步就算真正学会了。
