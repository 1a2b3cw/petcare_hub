# 第 11 步：开发阶段的页面性能优化

## 本次目标

解决一个很典型的开发期问题：**接了远程数据库以后，页面跳转明显变慢。**

这一步不追求做复杂性能工程，而是先用最适合当前项目阶段的办法，把体感优化到更合理的水平。

## 我做了什么

这次主要做了两件事：

1. 把下面三个高频页面从 `force-dynamic` 改成了 `revalidate = 30`
   - `src/app/(dashboard)/services/page.tsx`
   - `src/app/(dashboard)/customers/page.tsx`
   - `src/app/(dashboard)/customers/[id]/page.tsx`

2. 在宠物新增、编辑的 Server Action 里补上了列表页缓存刷新
   - `createPetAction`
   - `updatePetAction`

也就是原来只刷新客户详情页，现在会同时刷新：

- `/customers`
- `/customers/[id]`

## 为什么这样做

## 1. `force-dynamic` 会让每次访问都重新查库

之前这些页面写的是：

```ts
export const dynamic = "force-dynamic";
```

这表示每次请求都强制动态渲染，不走页面级缓存。

对于“本地开发 + 远程 Neon 数据库”这个组合来说，代价很明显：

- 你点一次页面，就要重新发一次数据库请求
- 数据库又不在本地，还会叠加网络延迟
- 如果当前是 `npm run dev`，Next.js 开发模式本来就更慢

所以慢并不一定是代码写坏了，很多时候是**渲染策略选得太激进**。

## 2. 这些页面并不需要强实时

当前项目里这些页面属于“读多写少”的后台页：

- 服务列表
- 客户列表
- 客户详情

它们不需要像聊天消息、股票行情那样每次都强制实时。

更合适的方案是：

- 页面平时允许短时间缓存
- 提交表单后手动 `revalidatePath`

这样能同时拿到两种好处：

- 普通跳转更快
- 新增/编辑后又能马上看到最新结果

## 3. 为什么选 `revalidate = 30`

这里故意没有设成很长，也没有直接完全静态化。

`30 秒` 是一个比较适合教学项目和后台系统演示的折中值：

- 足够减少频繁跨网络查库
- 又不会让缓存停留太久
- 配合 `revalidatePath()`，手动修改后基本还是即时更新

## 关键文件说明

### `src/app/(dashboard)/services/page.tsx`

服务列表页从“每次都实时查数据库”改成了“最多缓存 30 秒”。

因为服务新增、编辑、启停都已经在 action 里调用了：

```ts
revalidatePath("/services");
```

所以功能正确性不会受影响。

### `src/app/(dashboard)/customers/page.tsx`

客户列表页也改成了 `revalidate = 30`。

这个页面展示的不只是客户名和手机号，还展示：

- 宠物数量
- 宠物名称
- 最近完成服务

所以只刷新详情页是不够的，列表页也要能被主动刷新。

### `src/app/(dashboard)/customers/[id]/page.tsx`

客户详情页也改成了 `revalidate = 30`。

这样从列表点进详情时，不会每次都强制打一次数据库。

### `src/app/(dashboard)/customers/actions.ts`

这里是这一步最容易漏的点。

之前宠物新增/编辑只做了：

```ts
revalidatePath(`/customers/${customerId}`);
```

但现在列表页也有缓存，而且列表页会显示宠物信息，所以必须再补：

```ts
revalidatePath("/customers");
```

不然你在详情页新增了一只宠物，回到列表页可能会短时间看到旧的宠物数量。

## 你自己如何复现

如果你自己从零优化这一步，按这个顺序来：

1. 找出所有用了 `export const dynamic = "force-dynamic"` 的页面
2. 判断它是不是“必须每次实时查库”的页面
3. 对读多写少的页面改成：

```ts
export const revalidate = 30;
```

4. 回头检查所有相关的 Server Action
5. 确认写操作发生后，是否对所有受影响页面都执行了 `revalidatePath()`

验证方式：

```powershell
npm run dev
```

重点测试这几条路径：

1. `/services`
2. `/customers`
3. `/customers/[id]`

再测试这几个写操作：

1. 新增客户
2. 编辑客户
3. 新增宠物
4. 编辑宠物
5. 启停服务

确认写完后跳回去看到的是新数据，而不是旧缓存。

## 容易踩坑的点

1. **只改页面，不改 action**
   如果你把页面改成 `revalidate = 30`，但没有在写操作后主动刷新缓存，就会出现“我明明改成功了，页面怎么没变”的错觉。

2. **把所有页面都一刀切成缓存**
   有些页面仍然适合保持动态，比如编辑页、非常依赖最新数据的操作页。优化要先挑高频读页面，不要一上来全改。

3. **把开发模式速度当成线上真实速度**
   `npm run dev` 会比生产环境更慢。做性能判断时，最好再跑一次：

   ```powershell
   npm run build
   npm run start
   ```

   如果生产模式已经快很多，说明主要问题是开发模式和远程数据库叠加，而不是页面结构本身有大问题。

4. **忽略数据库地域**
   即使做了缓存，远程数据库的物理距离仍然会影响首次请求速度。
   所以最理想的组合仍然是：

   - 本地开发：本地 PostgreSQL
   - 线上演示：Neon

## 这一步做完后你应该得到什么

你现在的项目不再是“每进一次列表/详情就强制实时查远程数据库”。

它会变成：

- 平时读页面走短缓存
- 写操作完成后主动刷新对应页面
- 本地开发体感比之前更顺

这就是一个很典型的中后台项目优化思路：**不是盲目追求全实时，而是按业务场景选择缓存策略。**
