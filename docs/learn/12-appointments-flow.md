# 第 12 步：预约管理与状态流转

## 本次目标

把第三条核心业务主线真正接上数据库：

- 新建预约
- 查看预约列表
- 按状态筛选
- 在列表里推进预约状态

做到这一步后，项目里最关键的链路就真正连起来了：

**客户 / 宠物档案 → 选择服务 → 创建预约 → 门店确认 → 开始服务 → 完成服务**

## 我做了什么

这一轮新增和修改了这些文件：

```text
src/lib/validations/appointment.ts
src/app/(dashboard)/appointments/actions.ts
src/components/appointments/appointment-form.tsx
src/app/(dashboard)/appointments/new/page.tsx
src/app/(dashboard)/appointments/page.tsx
```

现在预约模块已经不是 mock 页面了，而是能直接操作数据库的真实页面。

## 这一步具体落地了哪些能力

### 1. 新建预约

新增了 `/appointments/new` 页面，表单里可以选择：

- 客户
- 宠物
- 服务项目
- 员工
- 日期
- 开始时间
- 备注

其中有两个关键联动：

1. 先选客户，再选该客户名下的宠物
2. 先选宠物，再筛出适用的服务项目

比如猫咪就不会给你展示只适用于狗狗的美容项目。

### 2. 预约自动算结束时间

表单里不要求你手动填结束时间。

系统会根据：

- 开始时间
- 服务项目时长

自动推算出 `endTime`，然后写进数据库。

这样做的好处是：

- 少一个输入项，录入更快
- 避免前台手动输错结束时间
- 以后做日历视图时，可以直接拿 `startTime` / `endTime` 排时间块

### 3. 预约编号自动生成

创建预约时，系统会按预约日期自动生成编号，例如：

```text
APT-20260419-001
```

这里的规则是：

- 日期部分取预约日期
- 当天已有多少条预约，就按顺序补 001 / 002 / 003

这个规则非常适合演示项目，既清楚又容易讲。

### 4. 预约列表接入真实数据库

`/appointments` 现在已经改成 Prisma 查询，不再是写死的 mock 数组。

列表会展示：

- 预约编号
- 客户 / 宠物
- 服务项目
- 开始时间 / 预计结束时间
- 员工
- 状态
- 操作按钮

### 5. 状态流转跑通

当前列表页支持最小可用的状态机：

```text
PENDING -> CONFIRMED -> IN_SERVICE -> COMPLETED
```

同时还支持：

```text
PENDING / CONFIRMED / IN_SERVICE -> CANCELLED
```

也就是说，门店现在已经可以在列表里完成这几个核心动作：

- 确认预约
- 开始服务
- 完成服务
- 取消预约

## 为什么这样做

## 1. 为什么不先做日历？

因为日历是“展示层增强”，不是主链路本身。

如果你先做日历，很容易把时间花在：

- 拖拽交互
- 时段布局
- 日历 UI 组件

但这些都不能证明“你的预约业务链真的跑通了”。

所以这一步故意先压缩范围，先把最核心的业务闭环做出来。

## 2. 为什么状态流转直接放在列表按钮里？

因为当前阶段最重要的是：

- 数据真实
- 流程能跑
- 代码易懂

如果现在就做成复杂弹窗、抽屉、二次确认框，复杂度会明显上升。

而列表按钮有个好处：**你点一下就能马上看到状态变化**，非常适合教学和联调。

## 3. 为什么宠物和服务要做联动筛选？

因为这已经属于业务正确性，而不是纯体验优化。

如果允许：

- 选张女士
- 再选别人的宠物
- 再选不适用当前宠物类型的服务

那预约虽然“能提交”，但数据已经脏了。

所以这一步既在前端做了筛选，也在服务端 action 里再次做了校验。

这叫做：

**前端负责减少误操作，服务端负责最终兜底。**

## 关键文件说明

### `src/lib/validations/appointment.ts`

这里把预约表单的字段校验统一收口了。

除了必填校验，还把可选字段做了处理：

- 空字符串会转成 `undefined`
- 备注最长 200 字

这样写库时更干净，不会把一堆空字符串塞进数据库。

### `src/app/(dashboard)/appointments/actions.ts`

这是本轮最关键的文件。

里面做了三类事情：

1. `createAppointmentAction`
   - 解析表单
   - 校验客户、宠物、服务、员工是否存在
   - 校验宠物是否属于当前客户
   - 校验服务是否适用于当前宠物
   - 自动计算 `endTime`
   - 自动生成 `appointmentNo`
   - 写入数据库

2. `advanceAppointmentStatusAction`
   - 根据当前状态推到下一个状态
   - `PENDING -> CONFIRMED -> IN_SERVICE -> COMPLETED`

3. `cancelAppointmentAction`
   - 把未完成的预约改成 `CANCELLED`

同时这几个 action 都会主动刷新：

- `/appointments`
- `/customers`
- `/customers/[id]`

因为客户页也会展示最近预约和最近完成服务，如果不一起刷新，页面就可能显示旧数据。

### `src/components/appointments/appointment-form.tsx`

这是一个客户端表单组件。

之所以这里用了 client component，不是为了炫技，而是因为它确实需要本地交互：

- 选中客户后联动宠物下拉
- 选中宠物后联动服务下拉
- 在表单底部即时显示已选服务的时长和价格

这类“表单局部交互”非常适合放客户端组件。

### `src/app/(dashboard)/appointments/page.tsx`

列表页从 mock 改成了真实查询，并加了状态筛选：

- 全部
- 待确认
- 已确认
- 服务中
- 已完成
- 已取消

筛选条件走 URL query（`?status=...`），而不是存在本地 Zustand 里。  
这样做的好处是：

- 刷新页面后筛选状态不会丢
- 可以直接复制带筛选条件的链接
- 服务器渲染时可以直接按条件查数据库

这比纯前端 store 更适合真实后台列表页。

## 你自己如何复现

如果你想自己不看答案重做一遍，可以按这个顺序：

1. 先补 `appointment.ts` 的 Zod schema
2. 再写 `appointments/actions.ts`
   - 新建预约
   - 推进状态
   - 取消预约
3. 再写 `AppointmentForm`
   - 客户 -> 宠物联动
   - 宠物 -> 服务联动
4. 新建 `/appointments/new/page.tsx`
   - 查客户 + 宠物
   - 查启用中的服务
   - 查可分配员工
5. 最后把 `/appointments/page.tsx` 从 mock 改成 Prisma 查询
6. 给每一行加状态操作按钮

验证时重点跑下面几步：

1. 创建一条新预约
2. 在列表里点“确认预约”
3. 再点“开始服务”
4. 再点“完成服务”
5. 去客户详情页看最近预约有没有同步变化

## 容易踩坑的点

1. **前端筛过一次，不代表服务端可以不校验**
   例如用户完全可以自己改请求参数，把 `customerId` 和 `petId` 换成不匹配的数据。
   所以 action 里必须再次验证。

2. **日期不要直接 `new Date("2026-04-19")`**
   这种写法会受到时区影响。当前实现用 `date-fns` 的 `parse()` 来按本地时间解析，更稳一些。

3. **价格字段不要直接把 Prisma Decimal 传给 client component**
   Server Component 往 Client Component 传值时，最好先转成普通 number，不然很容易踩序列化问题。

4. **改完预约状态，别忘了客户页也要刷新**
   因为客户页会展示最近预约和最近完成服务日期，这和预约状态直接相关。

到这一步，你的项目已经不只是“几个能看不能用的后台页面”了，而是真的有一条完整可演示的业务主线：

1. 先维护服务项目
2. 再维护客户和宠物档案
3. 然后创建预约
4. 最后推进预约状态直到完成

这已经很接近一个可以写进简历的中后台 MVP 了。
