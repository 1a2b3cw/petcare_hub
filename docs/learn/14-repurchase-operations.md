# 第 14 步：复购运营最小闭环

## 本次目标

把预约和履约之后的“运营动作”也接起来，先做一版足够轻、但业务上完整的复购闭环：

- 保存履约记录时自动生成回访任务
- 新增一个 `复购运营` 页面集中处理待回访
- 支持筛出沉睡客户

这样项目就不只是“能预约、能完成服务”，而是开始体现“服务完成后怎么把客户拉回来”的业务思考。

## 我做了什么

这一轮新增/修改了这些核心文件：

```text
src/app/(dashboard)/appointments/actions.ts
src/app/(dashboard)/operations/actions.ts
src/app/(dashboard)/operations/page.tsx
src/app/(dashboard)/dashboard/page.tsx
src/components/layout/app-shell.tsx
```

## 这一步新增了哪些能力

### 1. 保存履约记录时自动生成回访任务

现在你在预约详情页填写：

- 实际服务名称
- 服务记录
- 下次建议到店日期

并点击保存之后，系统会自动把 `nextSuggestedVisitAt` 同步成一条 `FollowUpTask`。

也就是说，履约记录不再只是“写在那里给人看”，而是会真正驱动后续的回访动作。

### 2. 新增 `复购运营` 页面

现在侧边栏多了一个页面：

```text
/operations
```

这个页面会集中展示：

- 待回访任务
- 已完成回访数
- 已跳过回访数
- 沉睡客户数量
- 未使用优惠券数量

### 3. 待回访任务可直接处理

在运营页里，每条待回访任务都可以：

- 标记完成
- 跳过

这意味着回访任务已经是一个可操作的业务对象，而不是只读列表。

### 4. 沉睡客户筛选

当前的沉睡客户定义是：

**最近一次已完成服务距离今天超过 30 天。**

这是一个很适合求职项目 MVP 的定义，因为：

- 简单
- 容易解释
- 能直接体现“老客复购”思路

## 为什么这样做

## 1. 为什么从履约记录自动生成 FollowUpTask？

因为在真实业务里，回访提醒往往不是运营同学手工一条条录进去的，而是来源于一次已经完成的服务。

把它接在履约记录之后，业务链会非常自然：

```text
预约完成 -> 填履约记录 -> 建议下次到店 -> 自动生成回访任务
```

这条链路比单独做一个“回访页面”更像真实系统。

## 2. 为什么先做回访提醒，不先做发券？

发券虽然也属于复购运营，但如果一上来就做：

- 发券表单
- 优惠类型
- 门槛金额
- 使用状态

会明显加重页面和交互。

而回访提醒更轻，也更能体现“客户运营”的基本逻辑，所以优先级更高。

## 3. 为什么沉睡客户放在同一页？

因为“待回访任务”和“沉睡客户筛选”本质上都是同一个问题：

**哪些客户该被重新联系了。**

把它们放一起更符合运营视角，也方便后面继续扩展，比如：

- 手动发券
- 批量创建回访任务
- 客户分层标签

## 关键文件说明

### `src/app/(dashboard)/appointments/actions.ts`

这里是本轮最关键的改动点。

在 `saveVisitRecordAction` 里，履约记录保存成功后，会继续调用一个同步函数，把：

- `visitRecord.id`
- `customerId`
- `petId`
- `actualServiceName`
- `nextSuggestedVisitAt`

同步成 `FollowUpTask`。

如果这个履约记录之前已经挂过回访任务，就更新；如果没有，就新建。

### `src/app/(dashboard)/operations/actions.ts`

这里放运营页的两个 Server Action：

- `completeFollowUpTaskAction`
- `skipFollowUpTaskAction`

并且会同步刷新：

- `/operations`
- `/dashboard`
- `/customers`
- 对应客户详情页

这样你在运营页点完按钮，其他页面的数据也能及时同步。

### `src/app/(dashboard)/operations/page.tsx`

这是新增的复购运营页。

页面拆成两块：

1. 左边：待回访任务表
2. 右边：沉睡客户卡片列表

顶部再加统计卡片，形成一个非常典型的后台运营页结构。

### `src/app/(dashboard)/dashboard/page.tsx`

工作台也从 mock 数字升级成了真实数据：

- 今日预约
- 待确认
- 今日完成
- 待回访

另外还会显示最近几条待处理回访，工作台终于开始像一个真实首页了。

## 你自己如何复现

如果你想自己重做这一轮，建议顺序是：

1. 先在预约 action 里补“履约记录 -> 回访任务”的自动同步
2. 再新建运营页 action，处理任务完成/跳过
3. 再做 `/operations/page.tsx`
4. 最后把侧边栏加上入口，并把工作台数据改成真实查询

## 验证方式

建议按下面路径完整走一遍：

1. 新建一条预约
2. 推进状态到 `服务中` 或 `已完成`
3. 在预约详情里填写“下次建议到店日期”
4. 保存履约记录
5. 打开 `/operations`，确认是否自动出现一条待回访任务
6. 点“标记完成”或“跳过”
7. 回到工作台确认待回访数字是否同步变化

## 容易踩坑的点

1. **只存 `nextSuggestedVisitAt`，不生成 `FollowUpTask`**
   这样数据虽然写进去了，但运营页永远没有任务，业务链就断了。

2. **运营页只刷新自己，不刷新工作台**
   这样你会看到任务处理了，但首页数字还是旧的，看起来像系统坏了。

3. **沉睡客户定义太复杂**
   这个阶段不要一上来就做多条件筛选。先用“30 天未完成服务”这个规则，把最小闭环跑通。

## 做完以后项目现在的状态

现在这套项目已经具备下面这条完整业务主线：

1. 维护服务项目
2. 维护客户与宠物
3. 创建预约
4. 推进预约状态
5. 填写履约记录
6. 自动生成回访任务
7. 在运营页处理回访与筛出沉睡客户

这已经不是普通的 CRUD 作品，而是开始具备“预约 + 履约 + 复购”的业务完整度了。
