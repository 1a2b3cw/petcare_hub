# 第 19 步：预约日历视图

## 本次目标

把预约模块从“只有列表页”升级成“列表 + 日历”双视图。

这样同一个模块就能满足两种典型使用场景：

- 列表视图：适合批量看状态、处理流程
- 日历视图：适合看当天排班、时段占用和空档

## 我做了什么

这一轮主要修改了：

```text
src/app/(dashboard)/appointments/page.tsx
```

## 这一步新增了哪些能力

### 1. 双视图切换

预约页现在支持：

- `列表视图`
- `日历视图`

并且是同一路由通过 query 参数切换，而不是拆成两个页面。

## 2. 日历视图按天查看

在日历视图里，可以直接：

- 看选中日期的预约安排
- 点“前一天 / 后一天”切换日期

这比单纯的预约列表更接近门店当天排班的使用方式。

### 3. 日历视图保留状态筛选

即使切到日历视图，顶部的状态筛选也还在。

也就是说你可以看：

- 某一天的全部预约
- 某一天的待确认预约
- 某一天的服务中预约

## 4. 时段式展示

日历视图当前采用的是**轻量时间轴**方案：

- 09:00 到 20:00 按小时分槽
- 每个时段下挂对应预约卡片
- 没有预约的时段显示“当前时段暂无预约”

这比真日历组件要轻很多，但已经足够表达“排班”这个核心信息。

## 为什么这样做

## 1. 为什么不直接上完整月历/周历组件？

因为完整日历虽然看起来更炫，但复杂度会明显上升：

- 月视图布局
- 周视图滚动
- 拖拽排班
- 冲突处理
- 时间块定位

当前项目阶段更适合先做一个**够用的日程视图**，把“当天排班能不能看明白”先解决。

## 2. 为什么列表和日历共用一个页面？

因为这两者本质上是在看同一批预约数据，只是展示方式不同。

共用一个页面有几个好处：

- URL 更统一
- 状态筛选逻辑可复用
- 不需要维护两套查询代码

## 关键文件说明

### `src/app/(dashboard)/appointments/page.tsx`

这一轮这里做了三类增强：

1. **搜索参数增强**
   增加了：
   - `view`
   - `date`

2. **查询逻辑增强**
   - 列表视图默认看全量历史预约
   - 日历视图按 `date` 过滤到当天
   - 两种视图都支持状态筛选

3. **展示层增强**
   - 顶部加视图切换
   - 日历视图加前一天 / 后一天切换
   - 增加“当天摘要”和“查看建议”辅助模块

## 你自己如何复现

如果你想自己从零做这一步，建议按下面顺序：

1. 先给预约页补 `view` 和 `date` 两个 query 参数
2. 再把 Prisma 查询改成根据视图决定是否按天过滤
3. 再写顶部视图切换和日期切换
4. 最后补日历视图的时间轴 UI

## 验证方式

建议这样测：

1. 打开 `/appointments`
2. 切到“日历视图”
3. 点“前一天 / 后一天”
4. 看预约是否跟着日期变化
5. 再切状态筛选，看日历视图是否还能正常过滤

## 容易踩坑的点

1. **列表和日历各写一套筛选逻辑**
   这样后面很容易一边改了、另一边忘了同步。

2. **日期参数不做兜底**
   如果 `date` 非法，页面就会炸。当前实现里做了安全解析，非法就退回今天。

3. **日历视图一上来做太重**
   当前项目目标是“可开发、可上线、可写进简历”，不是做一个通用日历库。先用轻量时间轴更划算。

## 做完以后项目状态

- 列表视图：更适合管理
- 日历视图：更适合排班

这会让你的项目从“后台 CRUD 页面”更进一步，开始有明显的产品体验层次。
