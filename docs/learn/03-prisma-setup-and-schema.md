# 03 Prisma 配置与数据库建模：先把数据底座立住

## 本次目标

这一步的目标不是“把数据库接上就行”，而是把项目的数据结构先定稳。

因为你这个项目本质上不是静态页面，而是一个中后台。只要是中后台，最终都要回到这些问题：

- 页面展示的是什么数据
- 数据之间是什么关系
- 业务状态怎么记录
- 哪些数据是后面要长期复用的

所以 Prisma 这一层，决定的是项目有没有真正的业务底座。

---

## 这一步我做了什么

### 1. 创建正式 Prisma 目录

我创建了：

- `prisma/schema.prisma`
- `prisma/seed.ts`
- `prisma.config.ts`

这表示项目已经从“只有草稿”进入“正式可执行的数据库工程状态”。

### 2. 把建模草稿落成正式 schema

前面我们写过 `PRISMA_SCHEMA_DRAFT.prisma`，那更像是设计草图。  
现在 `prisma/schema.prisma` 才是后续真正会参与：

- 生成 Prisma Client
- 创建迁移
- 推送数据库结构

的正式文件。

### 3. 根据业务主线建立核心模型

目前的核心模型包括：

- `User`
- `StoreProfile`
- `ServiceItem`
- `Customer`
- `Pet`
- `Appointment`
- `VisitRecord`
- `Coupon`
- `FollowUpTask`

这些模型基本就覆盖了你当前 MVP 的全部业务链路。

### 4. 为模型补了枚举

为了避免字段值混乱，我没有大量使用随意字符串，而是用枚举来固定状态和类型，比如：

- `UserRole`
- `AppointmentStatus`
- `CouponStatus`
- `FollowUpStatus`

这样做的好处是：

- 约束更强
- 前后端更容易统一
- 后续表单和筛选更好做

---

## 为什么要这样建模

## 1. 围绕“预约链路”而不是围绕“页面”建模

新手很容易按页面来想表结构，比如：

- 客户页一张表
- 预约页一张表
- 服务页一张表

这种想法只对了一半。更重要的是要看业务链路如何流动。

当前模型的核心逻辑是：

客户 -> 宠物 -> 服务项目 -> 预约 -> 履约记录 -> 复购动作

这条链一旦成立，页面只是把链路中的不同节点展示出来。

## 2. 为什么 `Appointment` 是核心表

因为你的项目名义上有很多模块，但真正最关键的是预约闭环。

`Appointment` 连接了：

- 客户
- 宠物
- 服务
- 员工
- 时间
- 状态

所以它是整个系统最重要的“中轴表”。

## 3. 为什么 `VisitRecord` 要独立成表

有人会想：既然预约完成后就结束了，为什么不把履约信息直接塞回 `Appointment`？

可以，但不推荐。

单独拆 `VisitRecord` 有两个好处：

1. 语义更清楚  
   预约是“计划”，履约记录是“真实发生”

2. 扩展性更好  
   后面你要补实际金额、宠物表现、下次建议到店时间，就不会把预约表塞得太乱

## 4. 为什么不做多门店

因为当前目标是学生独立可完成的作品集项目。

多门店一旦展开，马上会引入更多问题：

- 多租户数据隔离
- 角色权限更复杂
- 查询条件更多
- 运营口径更复杂

这会明显拖慢你完成主线的速度，所以现阶段砍掉是对的。

## Prisma 7 的一个重要变化

这次项目里有一个非常值得你记住的坑。

### 以前很多教程会这么写

在 `schema.prisma` 里这样写：

```prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

### 但这次不行

因为你现在装的是 Prisma 7，数据库连接串配置方式变了。  
如果继续把 `url` 写在 `schema.prisma` 里，执行 `prisma generate` 会直接报错。

### 现在的正确做法

把连接信息放到 `prisma.config.ts`：

```ts
import "dotenv/config";
import { defineConfig } from "prisma/config";

export default defineConfig({
  schema: "prisma/schema.prisma",
  migrations: {
    path: "prisma/migrations",
  },
  datasource: {
    url: process.env.DATABASE_URL,
  },
});
```

而 `schema.prisma` 里只保留：

```prisma
datasource db {
  provider = "postgresql"
}
```

### 你应该记住的结论

不是你的 schema 写错了，而是**Prisma 版本升级后，配置位置发生了变化**。

这类问题在真实开发里非常常见，所以你要学会看报错、查版本、做适配。

## 当前模型里你最该重点理解的部分

## 1. `ServiceItem`

它是预约时可被选择的服务项目。

关键字段包括：

- 名称
- 分类
- 时长
- 价格
- 适用宠物范围
- 是否启用

### 为什么有 `isActive`

因为服务项目通常不建议直接物理删除。  
如果它已经关联历史预约，删除会带来历史数据问题，所以更合理的是“停用”。

## 2. `Customer` 和 `Pet`

这里体现的是一对多：

- 一个客户可以有多只宠物

这样做符合真实门店场景，也为后面“客户详情下查看宠物列表”提供了天然支持。

## 3. `AppointmentStatus`

当前状态流转是：

- `PENDING`
- `CONFIRMED`
- `IN_SERVICE`
- `COMPLETED`
- `CANCELLED`

这组枚举是之后做预约操作按钮、状态标签、列表筛选的基础。

## 4. `FollowUpTask`

它不是营销系统，只是轻量回访任务。

这很符合当前 MVP 范围：先把“门店知道该回访谁”做出来，而不是一开始就接自动通知系统。

## 你自己如何复现

### 第一步：先写业务实体草稿

别急着直接敲 Prisma。先在脑子里确认这几个问题：

1. 核心业务链是什么
2. 哪些对象必须长期保存
3. 哪些状态是固定枚举

### 第二步：把草稿落成 `schema.prisma`

建议先建最核心的 5 类：

- 用户
- 服务
- 客户
- 宠物
- 预约

然后再往上加：

- 履约
- 优惠券
- 回访任务

### 第三步：给核心查询字段补索引

比如预约页一定经常查：

- 日期
- 状态
- 员工

那这些字段就应该考虑索引。

### 第四步：检查当前 Prisma 版本的配置方式

这是这次最值得你记住的一步。不要盲目照抄旧教程。

---

## 容易踩坑的点

## 1. 直接按页面建表

这会导致表结构只适合“当前页面展示”，不适合长期业务演进。

## 2. 过早做复杂抽象

比如一开始就做：

- 多门店
- 复杂角色系统
- 动态服务配置

这些都会让你主线延期。

## 3. 忽视 Prisma 版本差异

这次如果不处理 Prisma 7 的配置方式，后面你会卡在 `generate` 这一步。

## 4. 把预约和履约混成一张表

短期省事，长期会让数据语义变得模糊。
