# 09 第一个真实 CRUD：为什么先做服务项目管理

## 本次目标

这一步的目标是让项目第一次从“骨架页 + mock 数据”进入“真实业务模块”。

我这次选的不是预约，也不是客户，而是**服务项目管理**。  
原因很简单：

- 它最独立
- 它依赖最少
- 它最容易先做成闭环
- 它做完后能直接支撑预约模块

所以这是最适合当前阶段落地的第一个真实 CRUD。

---

## 我做了什么

这次主要完成了服务模块的第一版真实闭环：

1. 把 `services` 页面从 mock 数据改成真实 Prisma 查询
2. 新增了“新增服务”页面
3. 新增了“编辑服务”页面
4. 加上了“启用 / 停用”动作
5. 补了服务表单校验 schema
6. 补了最小表单 UI 组件

也就是说，这一轮虽然还没做“复杂弹窗”和“高级交互”，但已经具备了真正的：

- 读取
- 新增
- 更新
- 状态切换

---

## 为什么先做服务模块，而不是直接做预约

因为预约模块虽然更核心，但它依赖更多对象：

- 客户
- 宠物
- 服务
- 时间
- 员工
- 状态流转

如果你现在直接上预约，很容易一下子牵扯太多东西。

而服务模块只依赖 `ServiceItem` 这一张主表，是非常适合用来完成“第一个真实模块”的。

你可以把它理解成：

**先找一条最短路径，把“真数据闭环”跑通。**

---

## 这次的实现策略为什么这样选

## 1. 没有先上复杂弹窗

我这次用了：

- 列表页
- 新增页
- 编辑页

而不是一开始就做：

- Modal
- Drawer
- 乐观更新
- 多层前端状态同步

### 原因

你现在最重要的不是 UI 花样，而是：

1. 真正读到数据库
2. 真正写入数据库
3. 真正完成一次业务闭环

先把闭环做对，比先把交互做炫更重要。

## 2. 没有一开始就做物理删除

服务项目管理当前做的是：

- 新增
- 编辑
- 启停

没有做“删除”。

### 原因

这个项目里的服务项目未来会被预约和履约引用。  
如果直接删除，容易让历史数据失去意义。

所以这里更合理的策略是：

- 用 `isActive` 表示是否启用

这比简单删除更符合真实业务。

## 3. 表单校验先放服务端

这次我用 Zod 做表单校验，但先放在服务端动作里。

### 为什么

因为当前阶段最重要的是保证数据写入可靠。  
等后面你接 `React Hook Form` 时，可以再把客户端体验补上。

这也是一种常见开发顺序：

1. 先保证服务端数据正确
2. 再逐步增强客户端交互体验

## 这一步涉及了哪些关键文件

## 页面层

- `src/app/(dashboard)/services/page.tsx`
- `src/app/(dashboard)/services/new/page.tsx`
- `src/app/(dashboard)/services/[id]/edit/page.tsx`

### 作用

- 列表页负责读真实数据并展示
- 新增页负责创建服务
- 编辑页负责修改已有服务

## 动作层

- `src/app/(dashboard)/services/actions.ts`

### 作用

这里放服务模块的服务端动作：

- 创建服务
- 更新服务
- 切换启用状态

你可以把这里理解成“当前这个模块最轻量的后端入口层”。

## 校验层

- `src/lib/validations/service.ts`

### 作用

统一定义服务表单的输入规则，比如：

- 名称长度
- 时长范围
- 价格范围
- 分类枚举

## 组件层

- `src/components/services/service-form.tsx`
- `src/components/ui/input.tsx`
- `src/components/ui/textarea.tsx`
- `src/components/ui/submit-button.tsx`

### 作用

这些文件负责把新增和编辑表单复用起来，同时补齐项目当前还缺少的基础表单组件。

## 你自己如何复现

如果让你自己从零再做一遍这一轮功能，推荐顺序如下：

### 第一步：先把列表页接真实数据库

不要一上来就先做新增弹窗。  
先把 `services/page.tsx` 里的 mock 数据去掉，改成 Prisma 查询。

因为只要这一步通了，你就完成了第一个“真读”。

### 第二步：再做新增页

把“新增服务”单独做成一个页面，先完成最基础的表单和提交。

因为“新增页”是最简单的“真写”。

### 第三步：再抽表单组件

当你发现新增页和编辑页结构几乎一样时，再抽出 `ServiceForm`。

这比一开始就过度抽象更自然。

### 第四步：最后加启停

用 `isActive` 做启停，比删除更适合这个项目。

---

## 容易踩坑的点

## 1. 还没连上真实数据，就急着做交互细节

这会让你项目看起来很热闹，但核心价值没有提升。

## 2. 一开始就做删除

对服务项目这类会被历史记录引用的数据，删除通常不是最佳方案。

## 3. 没有复用新增和编辑表单

如果两份表单几乎一样，却各写一套，后面维护会变累。

## 4. 把所有逻辑堆进 `page.tsx`

服务端动作、校验、表单组件最好拆开，不然文件很快就会膨胀。
